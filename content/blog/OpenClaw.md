## Intro
점차 AI가 발전하면서 많은 사람들은 더 높은 편리성을 추구하고 있습니다.<br>
먼저 어떤 식으로 발전했는지 역사를 살펴보겠습니다.
``` python
규칙 기반 AI → 딥러닝 → Transformer → LLM → RAG → AI Agent → Multi Agent
```

이런 변화 속에서 제가 느낀 가장 큰 핵심 축은 **편리함**인 것 같습니다.<br>
편리함을 위해 **규칙 기반**에서 **거대 언어 모델**(LLM)이 탄생했고, 더 큰 자율성을 위해 LLM의 결과를 수행하는 **Agent**가 나왔습니다. 이에 그치지 않고, 자율성의 전문화를 위해 각 Agent의 역할을 나누어 실행하는 **Multi Agent**가 생겼습니다.

그리고 이런 자율성은 당연하게도 **권한**이라는 리스크가 따라옵니다.<br>
`"권한을 주다"`<br>
이 말이 저에게는 굉장히 폭력적이게 다가오는 것 같습니다.<br>
아마 해당 리스크를 다른 사용자들과 기업에서도 느끼기에, **OpenClaw**와 같은 자동화 AI 도구를 꺼리며 금지시킨다고 생각합니다.
<br>

<br>

그러나 앞서 말한 통상적인 생각과는 달리, 이 글에서는 **정반대**의 이야기를 해보려고 합니다.

> 만약 이런 자동화 AI를 활용해서 해킹의 'ㅎ'도 모르는 사람이 자연어로 해킹을 부탁했을 때, <br> 계획 수립부터 Script 생성, 공격 수행까지 가능하지 않을까?

이 질문의 답을 해보고자 연구를 진행해 봤습니다.

**OpenClaw**를 사용하여, 두 개의 시스템(Attacker, Victim)을 세팅하고, 자율 공격을 시뮬레이션하며 어느 정도까지 수행이 가능할까 확인해 보겠습니다.<br>
마지막에는 Insight를 공유하며 마쳐보도록 하겠습니다!

## Analysis

위 실험을 통해 나온 결과를 분석해보도록 하겠습니다.

확인해보고자 했던 **핵심**은 아래와 같습니다.
> 자연어 명령만으로 AI Agent가 어느 수준까지 공격 흐름을 이해하고, 계획하고, 실제 행위로 연결할 수 있는가?

결과적으로, OpenClaw는 단순히 명령어 하나를 실행하는 수준을 넘어, **사용자의 자연어 지시를 공격 단계로 분해 → 각 단계에 맞는 정찰 → 수집 → 보고**를 수행할 수 있음을 확인할 수 있었습니다.

특히 구체적인 명령어나 방법론이 없더라도 단계별 프로세스를 파악하고, 완벽하게 수행하였다는 점에서 사용자(공격자)의 **목적에 맞는 작업 단위로 재구성**할 수 있음을 볼 수 있습니다.

### 1. 자연어 기반 공격 자동화의 가능성

가장 먼저 확인된 점은, 사용자가 공격 절차를 자세히 알지 못하더라도 AI Agent가 이를 일정 수준 보완할 수 있다는 점입니다.

예를 들어 사용자는 “내부정찰”이라는 큰 범주의 행위만 지시했지만, OpenClaw는 사용자 정보, 시스템 정보, 계정 정보, 권한 그룹, 파일 탐색 등 여러 세부 작업으로 나누어 수행했습니다. 또한 각 행위를 MITRE ATT&CK 기법과 연결하여 보고했습니다.

이는 AI Agent가 단순 실행기가 아니라, 어느 정도의 보안 지식 기반을 바탕으로 “무엇을 먼저 확인해야 하는지” 판단할 수 있음을 보여줍니다.

특히 다음과 같은 점이 의미 있었습니다.

- 사용자가 구체적인 명령어를 제공하지 않아도 정찰 절차를 구성함
- 결과를 단순 출력하지 않고, 보안 관점에서 요약함
- 발견된 민감 파일을 후속 행위와 연결함
- 이후 추가 정찰 방향을 스스로 제안함

이러한 흐름은 기존 자동화 스크립트와 다른 특징입니다. 일반적인 스크립트는 미리 정의된 작업만 반복하지만, AI Agent는 현재 결과를 바탕으로 다음 행동을 선택하거나 제안할 수 있습니다.

<br>

### 2. Agent의 자율성은 “완전 자율”이 아니라 “조건부 자율”에 가까움

다만 이번 실험 결과를 “AI가 스스로 해킹을 완성했다”라고 해석하기에는 주의가 필요합니다.

이번 환경에서는 이미 Attacker와 Victim이 구성되어 있었고, C2 통신을 위한 서버와 에이전트도 사전에 준비되어 있었습니다. 즉, OpenClaw가 모든 공격 인프라를 처음부터 만든 것이 아니라, 준비된 실행 환경 위에서 자연어 명령을 받아 행위를 수행한 것입니다.

따라서 이번 실험에서 확인된 자율성은 다음과 같이 정리할 수 있습니다.

> OpenClaw는 공격 목적을 이해하고, 준비된 실행 권한과 통신 채널을 활용하여 정찰·수집·보고를 자동화할 수 있었다.

즉, OpenClaw의 능력은 LLM 자체만의 능력이 아니라, 다음 요소들이 결합된 결과입니다.

- 자연어를 해석하는 LLM
- 명령을 실제 시스템에서 수행할 수 있는 Agent 구조
- Attacker와 Victim 간 통신 채널
- 파일 다운로드 및 업로드 기능
- 결과를 다시 해석하고 다음 행동을 결정하는 루프

이 점이 중요합니다. LLM은 혼자서는 시스템에 접근하거나 명령을 실행할 수 없습니다. 하지만 실행 권한을 가진 Agent와 결합되면, 자연어 지시가 실제 시스템 행위로 이어질 수 있습니다.

결국 위험성은 “LLM이 똑똑하다”는 점보다, “LLM에게 실행 권한이 연결되었을 때 무엇이 가능한가”에 있습니다.

<br>

### 3. 민감 파일 탐색 및 수집 흐름

이번 실험에서 가장 직접적인 결과는 `API_key.txt` 파일의 탐색 및 수집이었습니다.

OpenClaw는 사용자 폴더 내에서 민감해 보이는 파일을 탐색했고, `Credential` 디렉터리 내부의 `API_key.txt`를 발견했습니다. 이후 해당 파일을 수집하여 Attacker 측 저장 경로에 저장했고, 추가 지시에 따라 localhost로 게시하는 흐름까지 수행했습니다.

이 과정은 실제 침해사고 관점에서 보면 다음 단계와 유사합니다.

1. 내부 파일 시스템 정찰
2. 민감 파일명 또는 확장자 기반 탐색
3. 자격 증명 또는 API Key 후보 식별
4. C2 채널을 통한 수집
5. 외부 접근 가능한 위치에 게시

중요한 점은, OpenClaw가 파일 내용을 깊게 이해하지 않아도 파일명, 경로, 디렉터리명만으로 민감도를 추정했다는 점입니다. `Credential`, `API_key`와 같은 이름은 공격자 입장에서 매우 높은 우선순위의 탐색 대상이 됩니다.

이는 방어 관점에서도 의미가 있습니다. 실제 환경에서 민감 파일이 사용자 바탕화면, 다운로드 폴더, 문서 폴더 등에 평문으로 존재한다면, 자동화 도구나 AI Agent에 의해 매우 쉽게 탐지될 수 있습니다.

따라서 이번 결과는 단순히 “OpenClaw가 파일을 가져왔다”는 수준이 아니라, 평문 자격 증명 관리의 위험성을 보여주는 사례이기도 합니다.

<br>

### 4. 추가 정찰 단계에서 확인된 행위

추가 프롬프트 이후 OpenClaw는 프로세스, 네트워크 연결, ARP, 라우팅 테이블, 자동실행 레지스트리, 최근 변경 파일 등으로 정찰 범위를 확장했습니다.

이는 공격자가 내부 시스템에서 다음과 같은 정보를 얻기 위해 수행하는 전형적인 흐름과 유사합니다.

- 현재 어떤 프로세스가 실행 중인지
- 외부와 연결된 네트워크 세션이 있는지
- 내부망 대역은 어떻게 구성되어 있는지
- 기본 게이트웨이와 라우팅 구조는 어떤지
- 자동실행 항목에 의심스러운 값이 있는지
- 최근 사용된 문서나 파일은 무엇인지

흥미로운 점은 OpenClaw가 추가 행위를 수행하면서도 “읽기 전용 정찰”이라는 표현을 사용했다는 점입니다. 이는 시스템 변경을 최소화하면서 정보를 수집하려는 방향으로 동작했음을 보여줍니다.

하지만 방어자 입장에서는 읽기 전용 행위라고 해서 안전하다고 볼 수 없습니다. 공격의 초기 단계에서 수행되는 대부분의 정찰 행위는 시스템을 직접 파괴하지 않지만, 이후 권한 상승, 내부 이동, 민감정보 탈취로 이어질 수 있는 기반 정보를 제공합니다.

따라서 이번 실험에서 수행된 정찰 행위들은 단독으로는 파괴적이지 않더라도, 공격 체인 전체에서는 중요한 전 단계라고 볼 수 있습니다.

<br>

### 5. OpenClaw의 한계

이번 실험에서 OpenClaw가 충분히 강력한 자동화 가능성을 보여주었지만, 동시에 몇 가지 한계도 확인할 수 있었습니다.

첫 번째는 환경 의존성입니다.

OpenClaw가 실제 행위를 수행하기 위해서는 이미 명령 실행이 가능한 Agent, C2 서버, 통신 구조가 준비되어 있어야 했습니다. 즉, 자연어 명령만으로 모든 공격이 가능한 것이 아니라, 실행 인프라가 갖춰져 있을 때 자동화 효과가 크게 나타납니다.

두 번째는 사용자 개입의 필요성입니다.

민감 파일을 수집한 뒤 localhost 업로드 단계에서 OpenClaw는 업로드 방식, 엔드포인트, 인증 여부 등을 확인하려고 했습니다. 이는 Agent가 완전히 독립적으로 모든 환경을 추론하는 것이 아니라, 불확실한 부분에서는 사용자에게 추가 정보를 요구한다는 의미입니다.

세 번째는 판단 기준의 단순성입니다.

민감 파일 탐색은 주로 파일명, 경로명, 확장자 기반으로 이루어졌습니다. 이는 빠르고 효과적이지만, 오탐과 미탐의 가능성이 존재합니다. 예를 들어 이름은 평범하지만 실제로는 민감한 파일이거나, 반대로 이름만 민감해 보이지만 실제 내용은 중요하지 않은 파일이 있을 수 있습니다.

네 번째는 공격 범위 통제 문제입니다.

사용자가 “계속 진행해줘”, “할 수 있는 행위를 수행해봐”와 같이 추상적인 명령을 내렸을 때, Agent가 어디까지 수행해야 하는지 경계가 모호해질 수 있습니다. 실험 환경에서는 통제된 VM이었기 때문에 문제가 되지 않았지만, 실제 운영 환경에서는 의도하지 않은 범위 초과 행위가 발생할 수 있습니다.

<br>

### 6. 보안적 시사점

이번 실험의 핵심 시사점은 다음과 같습니다.

> AI Agent의 위험성은 모델 자체보다, 모델에 연결된 권한과 실행 환경에서 발생한다.

LLM은 자연어를 해석하고 계획을 세울 수 있습니다. 여기에 시스템 명령 실행 권한, 파일 접근 권한, 네트워크 통신 기능이 결합되면, 자연어 지시가 실제 공격 행위로 이어질 수 있습니다.

따라서 AI Agent 기반 도구를 사용할 때는 다음과 같은 보안 통제가 필요합니다.

- Agent가 실행할 수 있는 명령 범위 제한
- 파일 접근 가능 경로 제한
- 외부 통신 목적지 제한
- 민감 파일 접근 시 승인 절차 적용
- 실행 로그 및 명령 이력 저장
- 비정상적인 정찰 행위 탐지
- C2 유사 통신 패턴 모니터링
- 사용자 자연어 프롬프트와 실제 실행 행위 간 매핑 기록

특히 기업 환경에서는 “AI가 무엇을 답변했는가”보다 “AI가 실제로 무엇을 실행했는가”를 기록하는 것이 중요합니다. Agent형 AI는 응답 텍스트보다 실행 결과가 더 큰 영향을 만들기 때문입니다.

<br>

### 7. 방어 관점에서의 탐지 포인트

이번 실험을 방어자 관점에서 보면 여러 탐지 포인트가 존재합니다.

우선 시스템 내부에서는 사용자 정보 조회, 시스템 정보 조회, 계정 목록 확인, 관리자 그룹 조회, 프로세스 목록 확인, 네트워크 연결 조회 등의 명령 실행 흔적이 남을 수 있습니다. 이러한 행위들은 단독으로는 정상 관리자 행위처럼 보일 수 있지만, 짧은 시간 안에 연속적으로 발생한다면 침해 초기 정찰 행위로 의심할 수 있습니다.

또한 민감 파일 탐색 과정에서는 사용자 프로필 하위 경로에 대한 반복적인 파일 검색이 발생할 수 있습니다. 특히 `credential`, `key`, `password`, `token`, `secret` 등과 관련된 파일명 탐색은 방어 측에서 중요한 탐지 기준이 될 수 있습니다.

네트워크 관점에서는 Victim과 Attacker 사이의 주기적인 통신, 파일 전송, localhost 또는 특정 포트로의 게시 행위가 관찰될 수 있습니다. 실험에서 사용된 Agent는 일정 주기와 jitter를 가진 통신 구조를 사용했기 때문에, 이러한 beaconing 형태의 트래픽 역시 탐지 대상이 될 수 있습니다.

즉, 이번 실험은 공격 자동화 가능성을 보여주는 동시에, 방어자가 어떤 로그와 행위를 주의 깊게 봐야 하는지도 함께 보여줍니다.

## Insight

본 글을 통해 제가 말씀드리고 싶은 부분은 크게 **3가지**입니다,

1. **다양한 시각에서 보자.**
  
    "권한을 주는 것은 로컬이 위험하다."라는 통념에서 벗어나서, 어떻게 활용할 수 있는지 유연하게 생각해보자! <br>AI가 발전하면서 좋은 아이디어만 있다면, 우리는 무엇이든 시도해 볼 수 있는 시대가 다가왔습니다. 이럴 때 가장 중요한 것은 창의력이 아닐까 생각합니다!

2. **AI는 발전할 것이고, 자동화는 더욱 고도화 될 것이다.**

    OpenClaw만으로도 이 정도의 가능성을 보여주었습니다. 현재 나온 Mythos부터, 앞으로 나오게 될 다양한 AI 모델들은 더 강한 성능으로서 더 큰 위협으로 작용할 것입니다.<br>성장이 급속도로 이뤄지는 만큼 개발, 보안 가릴 것 없이 다음을 예상하며 공부 및 업무를 하는게 중요하지 않을까 생각해봅니다.

3. **권한을 조심히 하자.**

    만들어지는 도구도 그렇고, 사용자들도 그렇고, 점점 권한을 준다는 행위에 무뎌지는 것 같습니다. 이는 말하고 있는 저 또한 마찬가지 입니다. <br>하지만 이런 편리함에는 언제나 리스크가 따르고, 우리는 그것을 항상 인지하면서 최소한의 권한으로 활용하는 자세가 필요하다고 생각합니다!
